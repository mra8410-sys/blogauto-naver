# 네이버블로그자동화 생성 지시문

이 문서는 코딩 에이전트가 `blogauto-naver`와 같은 시스템을 처음부터 구현하기 위한 **하네스 엔지니어링 기반 개발 지시문**이다.

목표는 기능 목록을 단순히 나열하는 것이 아니다. 목표는 다음 세 가지를 만족하는 자동화 하네스를 만드는 것이다.

1. 사용자의 지시를 계정, 카테고리, 키워드, 발행 목적, 이미지 스타일, 발행 옵션으로 구조화한다.
2. 여러 Codex 에이전트를 역할별로 분리하고, 각 에이전트 결과를 계약 JSON과 게이트로 검증한다.
3. 실패, 수정, 재검색, 재작성, 이미지 저장, 세션 만료, 중복 제목, 자동 발행 재시도를 루프 엔지니어링으로 제어한다.

코딩 에이전트는 이 문서를 보고 **완성 가능한 앱**을 만들어야 한다. 그러므로 모든 구현은 “입력 → 에이전트 실행 → 계약 검증 → 실패 분류 → 다음 루프 결정 → 이력 기록”의 하네스 구조 안에서 작성한다.

---

## 1. 최종 제품 정의

Windows 데스크톱 Electron 앱을 만든다.

앱은 네이버 블로그 글 생성과 발행을 자동화한다.

사용자는 다음을 설정한다.

- 네이버 계정
- 블로그 주소 ID
- 계정별 카테고리
- 카테고리별 검색 키워드
- 발행 목적
- 제외 주제
- 선호 톤
- 최신성 수준
- 이미지 생성 여부
- 본문 이미지 개수
- 공개/비공개
- 즉시/예약 발행
- 자동 반복 발행 주기
- Codex 에이전트별 모델 effort
- 계정별 샘플 이미지

앱은 다음 순서로 작업한다.

```text
계정/카테고리 선택
→ 필요 시 세션 확인
→ 필요 시 계정 샘플 이미지 스타일 분석
→ Research/Title Agent 실행
→ 필요 시 검색 후보 수집
→ Research/Title Agent 재실행
→ Writer Agent 실행
→ Main Agent 최종 검수
→ 필요 시 Writer 재시도
→ Image Worker 실행
→ 이미지 파일 정규화/저장
→ 제목 중복 검사
→ 미리보기
→ 네이버 글쓰기 입력
→ 발행
→ 이력 기록
```

## 2. 하네스 엔지니어링 원칙

이 시스템의 중심은 UI도, 네이버 자동화도, Codex 프롬프트도 아니다.

중심은 **하네스**다.

하네스는 다음 책임을 가진다.

- 작업 상태를 하나의 상태기계로 관리한다.
- 모든 에이전트 입력과 출력을 파일로 남긴다.
- 모든 에이전트 결과를 JSON 계약으로 검증한다.
- 결과 상태를 `PASS`, `REVISION`, `BLOCK`, `failed`, `success` 등 명확한 상태로 분류한다.
- 실패가 어느 단계에서 발생했는지 `failurePhase`로 기록한다.
- 재시도 가능한 실패와 즉시 중단해야 하는 실패를 분리한다.
- 자동 발행 대상 순회와 재시도 횟수를 제어한다.
- 토큰 사용량을 agent별, gross/effective별로 합산한다.
- 사용자가 볼 로그와 내부 디버그 파일을 분리한다.
- 로컬 계정 정보와 빌드 산출물이 Git에 올라가지 않도록 차단한다.

코딩 에이전트는 기능을 만들 때마다 다음 질문을 적용한다.

1. 이 기능의 입력 계약은 무엇인가?
2. 출력 계약은 무엇인가?
3. 실패는 어떤 상태로 분류되는가?
4. 재시도 가능한가?
5. 재시도한다면 어느 루프가 담당하는가?
6. 사용자에게 보여줄 로그는 무엇인가?
7. job 폴더에는 어떤 증거 파일을 남기는가?
8. `npm run check`로 어떤 회귀를 막을 것인가?

## 3. 루프 엔지니어링 원칙

이 시스템에는 여러 종류의 루프가 있다.

코딩 에이전트는 구현 시 루프를 명시적으로 설계해야 한다.

### 3.1 개발 루프

각 모듈은 다음 순서로 만든다.

```text
계약 정의
→ 최소 구현
→ 정적 검사 추가
→ 실패 케이스 작성
→ 정상 케이스 작성
→ 통합 흐름 연결
→ 로그와 이력 확인
→ npm run check 통과
```

새 기능을 추가할 때는 먼저 하네스 계약과 검증 지점을 정한다.

### 3.2 에이전트 루프

에이전트는 단독으로 성공을 선언할 수 없다.

앱 하네스가 다음을 확인한다.

- JSON 파싱 가능 여부
- 필수 필드 존재 여부
- 상태값 유효성
- 실패 사유 존재 여부
- 다음 단계로 넘겨도 되는지
- 재시도해야 하는지
- 검색, 재작성, 중단 중 무엇을 선택해야 하는지

### 3.3 Research 검색 루프

Research/Title Agent는 직접 웹을 탐색하지 않는다.

검색이 필요하면 `searchNeed`를 반환하고 앱이 Search Collector를 실행한다.

```text
Research 1차 판단
→ searchNeed 확인
→ light/normal/strict이면 검색 수집
→ sourceQuality 계산
→ Research 재분석
→ PASS/REVISION/BLOCK 분기
```

### 3.4 Writer/Main 수정 루프

Writer Agent는 글을 쓰고, Main Agent는 검수한다.

자동 주제 모드에서는 최대 3회 루프를 돈다.

```text
Writer 작성
→ Writer 구조 검증
→ Main 최종 검수
→ PASS이면 종료
→ REVISION 또는 PASS 검증 실패이면 피드백 생성
→ Writer 재작성
→ 최대 횟수 초과 시 main_review 실패
```

수동 주제 모드는 기본 1회만 실행한다.

### 3.5 이미지 저장 루프

Image Worker가 로컬 파일 경로를 주면 그대로 복사한다.

파일 경로가 없고 Codex 세션에 base64 이미지 결과가 있으면 앱이 세션 JSONL을 읽어 파일로 저장한다.

```text
Image Worker 실행
→ 로컬 path 확인
→ path 없으면 Codex session image_generation 결과 검색
→ base64 decode
→ jobDir/recovered-images 저장
→ runtime/image 복사
→ 미리보기 URL 생성
```

이 흐름은 실패가 아니라 정상 저장 경로로 취급한다.

### 3.6 자동 발행 루프

자동 발행은 계정/카테고리 조합을 순회한다.

```text
대상 선택
→ 세션 상태 확인
→ 작업 실행
→ 성공이면 다음 대상
→ Research 실패면 최대 2회 재시도
→ 일반 실패면 최대 3회 재시도
→ 세션 만료면 세션 확인 또는 반복 주기까지 대기
→ Codex 사용량 한도면 전체 중단
```

## 4. 상태기계

작업 상태는 최소 다음을 지원한다.

```text
idle
generating
publishing
generated
success
failed
duplicate_retry
session_expired
codex_usage_limit
```

상태 전이는 하네스에서만 수행한다.

예시:

```text
idle → generating
generating → generated
generating → duplicate_retry
generating → failed
generating → codex_usage_limit
generated → publishing
publishing → success
publishing → session_expired
publishing → failed
```

실패에는 반드시 `failurePhase`를 붙인다.

허용되는 대표 failurePhase:

```text
research
writer
main_review
publish
session
image
```

## 10. Codex Runner 하네스

`src/lib/codexRunner.js`는 시스템의 핵심이다.

책임:

- Codex CLI 실행
- 에이전트별 프롬프트 파일 작성
- 에이전트별 결과 JSON 읽기
- JSON 파싱 실패 요약
- 토큰 사용량 추적
- rate limit 추적
- 에이전트 간 루프 제어
- 실패 phase 분류

각 작업은 job 폴더에 다음 파일을 남긴다.

```text
image-style-prompt.txt
image-style-result.json
research-title-initial-prompt.txt
research-title-initial-result.json
research-title-prompt.txt
research-title-result.json
prompt.txt
prompt-retry-2.txt
prompt-retry-3.txt
agent-result.json
main-review-prompt.txt
main-review-retry-2.txt
main-review-retry-3.txt
main-review-result.json
image-worker-prompt.txt
image-worker-result.json
```

Codex stdout에서 다음 progress를 감지한다.

```text
BLOGAUTO_PROGRESS: research
BLOGAUTO_PROGRESS: title
BLOGAUTO_PROGRESS: source_review
BLOGAUTO_PROGRESS: date_filter
BLOGAUTO_PROGRESS: article
BLOGAUTO_PROGRESS: main_review
BLOGAUTO_PROGRESS: image
BLOGAUTO_PROGRESS: save
```

토큰 사용량은 다음 값을 보존한다.

```json
{
  "total": 0,
  "grossTotal": 0,
  "inputTokens": 0,
  "cachedInputTokens": 0,
  "outputTokens": 0,
  "agents": {
    "main": 0,
    "research": 0,
    "writer": 0,
    "image": 0,
    "imageStyle": 0

  "grossAgents": {
    "main": 0,
    "research": 0,
    "writer": 0,
    "image": 0,
    "imageStyle": 0
  },
  "rateLimits": null
}
```

`total`은 캐시 입력 토큰을 제외한 실효 토큰 기준이다.

## 12. Image Style Agent 하네스

### 목적

계정별 샘플 이미지를 분석해 이미지 생성 스타일 프롬프트를 만든다.

### 입력

```json
{
  "sampleImagePath": "",
  "sampleImageHash": "",
  "jobDir": ""
}
```

### 출력 계약

```json
{
  "status": "success | failed",
  "failureReason": "",
  "imageStylePrompt": "",
  "sampleImageHash": "",
  "notes": []
}
```

### 하네스 루프

```text
sampleImagePath 존재 확인
→ imageStylePrompt 존재 확인
→ imageStylePromptStatus 확인
→ sampleImageHash/sourceHash 비교
→ 필요 시 Image Style Agent 실행
→ 결과 JSON 검증
→ success + prompt 있으면 계정 저장소 ready
→ 실패하면 failed 저장 후 기본 스타일로 계속
```

### 게이트

- 이미지가 없으면 실행하지 않는다.
- 프롬프트가 없으면 실행한다.
- 상태가 `missing`, `stale`, `failed`이면 실행한다.
- 이미지 해시가 바뀌었으면 실행한다.
- 실패해도 전체 글 생성은 멈추지 않는다.

## 13. Research/Title Agent 하네스

### 목적

주제를 선정하고, 제목을 확정하고, 검색 필요성을 판단하고, Writer가 따라야 할 compact writerContract를 만든다.

Research/Title Agent는 본문을 쓰지 않는다.

### 입력

```json
{
  "topic": "",
  "keyword": "",
  "category": "",
  "topicMode": "manual | auto",
  "excludedTopics": "",
  "publishPurpose": "",
  "preferredTone": "",
  "freshnessLevel": "auto | low | medium | high",
  "historyTitles": [],
  "searchResults": [],
  "sourceQuality": {}
}
```

### 출력 계약

### searchNeed 규칙

- `skip`: 안정적 개념, 설명, 의견, 경험형 주제
- `light`: 가벼운 검색 확인 필요
- `normal`: 검색 후보와 본문 확인 필요
- `strict`: 공식성, 현재성, 신청/모집/정책/가격/일정/마감일 검증 필요

### Research 루프

```text
Research 1차 실행
→ searchNeed 유효성 검사
→ skip이면 sourceQuality skipped
→ light/normal/strict이면 onSearchNeeded 호출
→ 검색 후보 수집
→ sourceQuality 계산
→ Research 2차 실행
→ status 게이트
→ finalTitle 게이트
→ writerContract 게이트
```

### 게이트

- `searchNeed`가 네 값 중 하나가 아니면 `failurePhase: research`.
- 검색 필요 상태인데 검색 결과가 없으면 `failurePhase: research`.
- `BLOCK`이면 본문 작성 금지.
- `REVISION`이면 본문 작성 금지.
- `PASS`라도 `finalTitle`이 비어 있으면 실패.
- writerContract는 검색 과정을 설명하지 말고 작성 계약만 담아야 한다.

## 15. Writer Agent 하네스

### 목적

Research/Title Agent가 확정한 제목과 writerContract에 따라 실제 블로그 본문을 작성한다.

Writer는 새 주제를 만들지 않는다.

Writer는 제목을 바꾸지 않는다.

Writer는 리서치 과정을 독자에게 설명하지 않는다.

### 출력 계약

```json
{
  "status": "success | failed",
  "failureReason": "",
  "title": "",
  "article": "",
  "tags": [],
  "titleImagePath": "",
  "titleImagePrompt": "",
  "bodyImages": [
    {
      "sequence": 1,
      "path": "",
      "prompt": ""
    }
  ],
  "notes": []
}
```

### 작성 규칙

- 한국어로 쓴다.
- `[SECTION - 제목]` 마커를 사용한다.
- 첫 섹션은 독자가 얻을 내용을 바로 설명한다.
- “이 글은 일반 조언이 아니라” 같은 방어적 도입을 쓰지 않는다.
- 발행 목적 문장을 본문에 복사하지 않는다.
- factBased 주제는 confirmedFacts와 usableSources 범위 안에서만 쓴다.
- 불확실한 항목은 단정하지 않는다.
- 오늘 날짜, 작성일, 현재 날짜를 불필요하게 노출하지 않는다.
- 출처 문장을 그대로 복사하지 않는다.
- 실제 이미지 생성은 하지 않고 프롬프트만 작성한다.

### Writer 구조 검증 루프

```text
Writer 실행
→ JSON 파싱
→ status 확인
→ article 비어 있음 확인
→ tags 배열 확인
→ 날짜/기준일 누출 확인
→ retryable writer failure 판단
→ 성공이면 Main Agent로 전달
```

### 재시도 규칙

- 자동 주제 모드에서는 Writer/Main 루프 안에서 최대 3회 시도한다.
- 수동 주제 모드에서는 기본 1회 시도한다.
- Writer 실패가 날짜 누출, 소스 부족, Research revision 연동 문제이면 재시도 가능하다.
- 재시도 시 `writerRevisionFeedback`을 프롬프트에 넣는다.
- 재시도 Writer는 기존 finalTitle과 writerContract를 유지해야 한다.

## 16. Main Agent 검수 하네스

### 목적

Writer 결과가 발행 가능한지 독립적으로 검수한다.

Writer의 `status: success`를 신뢰하지 않는다.

### 출력 계약

```json
{
  "status": "PASS | REVISION | BLOCK",
  "failureReason": "",
  "titleReviewPass": true,
  "articleAnswersTitle": true,
  "topicPreserved": true,
  "factualityPass": true,
  "sourceUsePass": true,
  "bodyQualityPass": true,
  "riskExpressionPass": true,
  "writerContractPass": true,
  "readerFacingArticlePass": true,
  "noResearchProcessNarrationPass": true,
  "publishable": true,
  "issues": [],
  "revisionInstructions": [],
  "notes": []
}
```

### 검수 기준

- finalTitle과 Writer 제목이 일치해야 한다.
- 본문이 제목에 답해야 한다.
- topicThesis가 보존되어야 한다.
- writerContract의 mustAnswer, mustCover, mustNotDo를 지켜야 한다.
- factBased 주제는 Research가 넘긴 사실 범위 안에서만 작성되어야 한다.
- 현재성 검증이 필요한 주제는 기준, 신청 경로, 대상, 조건 등을 확인해야 한다.
- 공식 또는 신뢰 가능한 출처가 필요한데 없으면 PASS할 수 없다.
- 도입부가 리서치 보고서처럼 보이면 REVISION.
- 발행 목적을 복사하면 REVISION.
- `[SECTION - ...]` 마커는 허용한다.

### PASS 검증 루프

Main Agent가 PASS를 반환해도 앱이 다시 검증한다.

```text
Main Agent 실행
→ status 확인
→ PASS이면 모든 boolean true 확인
→ failureReason 비어 있는지 확인
→ publishable true 확인
→ 하나라도 실패하면 PASS verification failed
→ 재시도 가능하면 Writer 재실행
→ 재시도 불가하면 failurePhase main_review
```

### Writer/Main 루프

```text
Writer 작성
→ Writer 구조 검증
→ Main 검수
→ PASS 검증 통과 시 종료
→ REVISION이면 feedback 생성
→ PASS 검증 실패도 feedback 생성
→ Writer 재작성
→ 최대 시도 초과 시 실패
```

- imageAssets가 Codex session image_generation base64를 복구하는지
- publishButton selector가 `data-click-area='tpb.publish'`를 우선하는지
- AI 활용 설정 helper가 `.se-set-ai-mark-button-toggle.se-is-selected`를 인식하는지
- 자동 발행이 Research 실패 2회, 일반 실패 3회 규칙을 갖는지
- runtime/dist/env/exe/account JSON ignore 규칙이 있는지

## 27. 최종 완료 기준

다음을 모두 만족해야 한다.

1. `npm install` 후 `npm start`가 동작한다.
2. `npm run check`가 통과한다.
3. 계정, 카테고리, 샘플 이미지 관리가 된다.
4. 계정별 이미지 스타일 프롬프트가 생성/저장/초기화된다.
5. Research/Title Agent가 searchNeed를 판단하고 검색 루프를 돈다.
6. Search Collector가 strictEvidence를 조건부로 적용한다.
7. Writer Agent는 writerContract를 기준으로만 본문을 쓴다.
8. Main Agent는 PASS boolean 전체를 검증한다.
9. Writer/Main 루프가 자동 주제 모드에서 최대 3회 돈다.
10. Research 실패는 자동 발행에서 최대 2회만 재시도한다.
11. 일반 실패는 자동 발행에서 최대 3회 재시도한다.
12. Image Worker는 Codex 이미지 생성을 사용한다.
13. 세션 base64 이미지는 파일로 저장된다.
14. 이미지 경로 없음은 실패 로그가 아니라 파일 저장 흐름으로 표시된다.
15. 네이버 발행 버튼은 `data-click-area='tpb.publish'`를 우선 사용한다.
16. AI 활용 설정 토글을 찾아 켠다.
17. 제목 유사도 0.75 이상이면 발행하지 않는다.
18. 세션 만료 계정은 자동 발행에서 대기/재개 흐름을 탄다.
19. Codex 사용량 한도는 전체 자동 발행을 중단한다.
20. runtime, dist, env, exe, 계정 JSON은 Git에 올라가지 않는다.

## 28. 코딩 에이전트의 작업 방식

코딩 에이전트는 한 번에 모든 것을 만들려고 하지 말고, 하네스 루프를 작은 단위로 완성한다.

각 단계마다 다음을 수행한다.

```text
1. 계약 작성
2. 구현
3. 실패 분류
4. 로그 작성
5. 이력 또는 산출 파일 작성
6. check 규칙 추가
7. npm run check
8. 다음 단계로 이동
```

기능이 보이더라도 하네스 계약이 없으면 미완성으로 본다.

프롬프트가 있어도 결과 검증이 없으면 미완성으로 본다.

자동화가 작동해도 이력과 실패 분류가 없으면 미완성으로 본다.

발행이 성공해도 보안 제외 규칙이 없으면 미완성으로 본다.

이 원칙을 지키면 코딩 에이전트는 현재 시스템과 유사한 구조의 네이버 블로그 자동화 앱을 재현할 수 있다.
