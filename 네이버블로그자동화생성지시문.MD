# 네이버 블로그 자동화 시스템 생성 지시문

이 문서는 코딩 에이전트가 처음부터 현재 수준의 네이버 블로그 자동화 데스크톱 앱을 구현하기 위한 개발 지시문이다. 구현 언어와 런타임은 Electron 기반 Node.js 앱을 기준으로 한다.

## 1. 목표

네이버 블로그 글쓰기와 발행을 자동화하는 Windows 데스크톱 앱을 만든다.

사용자는 앱에서 네이버 계정, 블로그 ID, 카테고리, 검색 키워드, 발행 목적, 공개 범위, 예약 발행 여부, 이미지 생성 여부를 설정한다.

앱은 Codex 에이전트를 여러 역할로 나누어 실행한다.

- Image Style Agent: 계정별 샘플 이미지를 분석해 이미지 스타일 프롬프트를 만든다.
- Research/Title Agent: 주제와 제목을 선정하고 검색 필요성을 판단한다.
- Search Collector: Research/Title Agent가 요청한 경우 외부 검색 후보를 수집한다.
- Writer Agent: 확정된 제목과 writerContract를 기준으로 본문과 이미지 프롬프트를 작성한다.
- Main Agent: Writer 결과를 최종 검수하고 PASS, REVISION, BLOCK을 판단한다.
- Image Worker: Writer가 만든 이미지 프롬프트로 Codex 이미지 생성을 수행한다.
- Publisher: Chrome과 Playwright를 이용해 네이버 블로그 글쓰기 화면에 입력하고 발행한다.

## 2. 기술 스택

- Electron을 사용해 Windows 데스크톱 앱을 만든다.
- Node.js CommonJS 모듈을 사용한다.
- Playwright Core로 Chrome을 제어한다.
- Codex CLI 또는 `codex.cmd`를 child process로 실행한다.
- 설정과 작업 결과는 로컬 `runtime/` 폴더에 JSON, JSONL, 이미지 파일로 저장한다.
- 앱 빌드는 `electron-builder`의 Windows portable target을 사용한다.

## 6. 계정과 카테고리 모델

계정 저장소 파일은 `runtime/account-categories.json`이다.

기본 형태는 다음과 같다.

```json
{
  "selectedAccountId": "",
  "accounts": []
}
```

계정 객체는 다음 필드를 가진다.

```json
{
  "id": "acct_timestamp_random",
  "label": "표시 이름",
  "naverId": "네이버 로그인 ID",
  "blogId": "블로그 주소 ID",
  "naverPassword": "비밀번호",
  "sampleImagePath": "",
  "sampleImageHash": "",
  "sampleImageUpdatedAt": "",
  "imageStylePrompt": "",
  "imageStylePromptUpdatedAt": "",
  "imageStylePromptStatus": "missing | ready | stale | failed",
  "imageStylePromptSourceImageHash": "",
  "imageStylePromptError": "",
  "checked": true,
  "sessionStatus": "unknown | valid | expired",
  "sessionCheckedAt": "",
  "categories": []
}
```

카테고리 객체는 다음 필드를 가진다.

```json
{
  "id": "cat_timestamp_random",
  "name": "카테고리명",
  "keyword": "검색 키워드 목록",
  "excludedTopics": "제외 주제",
  "publishPurpose": "발행 목적",
  "preferredTone": "선호 톤",
  "freshnessLevel": "auto | low | medium | high",
  "checked": true
}
```

계정 샘플 이미지가 등록되면 `runtime/account-assets/<accountId>/sample.<ext>`로 복사한다.

이미지를 변경하면 해시를 갱신하고 `imageStylePromptStatus`를 `stale` 또는 `missing`으로 바꾼다.

이미지를 삭제하면 이미지 경로, 해시, 스타일 프롬프트, 상태, 오류를 모두 초기화한다. 이 경우 기본 이미지 스타일로 진행한다.

## 7. UI 요구사항

앱 첫 화면은 실제 작업 콘솔이어야 한다.

반드시 포함할 영역은 다음과 같다.

- 상단 상태바
- 누적 토큰 배지
- Codex 5시간, 주간 사용량 배지
- Main, Research/Title, Writer, Image Worker 모델 effort 선택
- 계정 목록과 계정 등록, 수정, 삭제, 세션 확인
- 계정별 샘플 이미지 미리보기, 업로드/변경, 삭제
- 카테고리 목록과 카테고리 등록, 수정, 삭제
- 발행 목적, 검색 키워드, 제외 주제, 선호 톤, 최신성 수준
- 발행 옵션: 생성 후 발행, 공개 범위, 즉시/예약 발행
- 작업 로그
- 선정 제목 표시
- 본문 미리보기
- 이미지 미리보기와 이미지 처리 노트
- 자동 발행 시작과 정지

자동 발행 모드에서는 체크된 계정과 체크된 카테고리 조합만 대상으로 삼는다.

세션 만료 계정은 즉시 실패시키지 말고 세션 확인 또는 반복 주기까지 대기한다.

## 8. 전체 작업 흐름

단일 작업의 흐름은 다음 순서로 구현한다.

1. 폼과 선택 계정, 카테고리를 수집한다.
2. 발행이 필요한 경우 네이버 ID가 있는지 확인한다.
3. 계정별 브라우저 프로필 경로를 결정한다.
4. 발행 전 세션 확인이 필요한 경우 Chrome을 열어 네이버 글쓰기 편집기 접근 가능성을 확인한다.
5. job 폴더를 만든다.
6. 계정 샘플 이미지가 있고 스타일 프롬프트가 없거나 stale이면 Image Style Agent를 먼저 실행한다.
7. Research/Title Agent를 실행한다.
8. Research/Title Agent가 검색을 요청하면 Search Collector를 실행하고 Research/Title Agent를 재실행한다.
9. Research/Title Agent가 PASS이고 최종 제목이 있으면 Writer Agent를 실행한다.
10. Writer Agent 결과를 Main Agent가 검수한다.
11. Main Agent가 REVISION이면 Writer Agent를 다시 실행한다.
12. Main Agent가 PASS하면 Image Worker를 실행한다.
13. 이미지 결과를 정규화하고 세션 base64 이미지가 있으면 파일로 저장한다.
14. 제목 중복 검사를 수행한다.
15. 미리보기를 렌더러에 전달한다.
16. 발행 옵션이 켜져 있으면 네이버 블로그에 입력하고 발행한다.
17. 작업 이력을 `blog_history.jsonl`에 기록한다.

## 9. Codex 실행 공통 규칙

각 에이전트는 별도 프롬프트 파일과 결과 JSON 파일을 job 폴더에 쓴다.

Codex 실행은 다음 원칙을 따른다.

- 에이전트별 effort를 선택할 수 있어야 한다.
- stdout에서 `BLOGAUTO_PROGRESS: 단계명`을 감지해 UI 상태를 갱신한다.
- Codex 결과 JSON 파싱 실패는 원문 전체를 노출하지 말고 요약 오류로 처리한다.
- 토큰 사용량은 세션 로그와 Codex 출력에서 복구한다.
- 누적 토큰은 캐시 입력 토큰을 제외한 실효 토큰을 기준으로 계산한다.
- gross 토큰, cached input, input, output, agent별 합계도 별도로 보존한다.
- rate limit 정보가 있으면 runtime에 저장하고 UI 배지로 표시한다.
- Codex usage limit 오류는 일반 실패가 아니라 `codex_usage_limit` 상태로 자동 작업을 중단한다.

## 10. Image Style Agent

### 역할

계정에 등록된 샘플 이미지를 분석해 계정별 이미지 스타일 프롬프트를 만든다.

### 입력

- sampleImagePath
- sampleImageHash
- jobDir

### 출력 JSON

```json
{
  "status": "success | failed",
  "failureReason": "",
  "imageStylePrompt": "이미지 스타일 지시문",
  "sampleImageHash": "원본 이미지 해시",
  "notes": []
}
```

### 루프 검증 단계

1. 작업 시작 시 계정에 `sampleImagePath`가 있는지 확인한다.
2. 이미지가 없으면 Image Style Agent를 실행하지 않는다.
3. 이미지가 있고 `imageStylePrompt`가 비어 있으면 실행한다.
4. `imageStylePromptStatus`가 `missing`, `stale`, `failed`이면 실행한다.
5. `imageStylePromptSourceImageHash`와 현재 `sampleImageHash`가 다르면 실행한다.
6. 결과가 success이고 `imageStylePrompt`가 비어 있지 않으면 계정 저장소에 저장하고 상태를 `ready`로 바꾼다.
7. 결과가 failed이거나 프롬프트가 비어 있으면 상태를 `failed`로 저장하되, 전체 작업은 중단하지 않고 기본 이미지 스타일로 계속한다.

## 11. Research/Title Agent

### 역할

카테고리, 키워드, 발행 목적, 제외 주제, 최신성 수준을 바탕으로 발행 가능한 제목과 writerContract를 만든다.

Research/Title Agent는 본문을 쓰지 않는다.

### searchNeed 판단

`searchNeed`는 반드시 다음 중 하나여야 한다.

- `skip`: 안정적 개념 설명, 의견, 경험형 글이라 현재 검색이 없어도 안전한 경우
- `light`: 가벼운 검색 확인이 필요한 경우
- `normal`: 일반 검색과 후보 본문 확인이 필요한 경우
- `strict`: 공식성, 현재성, 신청 가능성, 모집 여부, 정책 조건처럼 강한 출처 검증이 필요한 경우

### 출력 JSON

다음 구조를 반환해야 한다.

```json
{
  "status": "PASS | REVISION | BLOCK",
  "failureReason": "",
  "finalTitle": "",
  "topicThesis": "",
  "directTopicPreserved": true,
  "factBased": true,
  "searchNeed": "skip | light | normal | strict",
  "searchFlowSummary": "",
  "repeatedTopics": [],
  "competitionGaps": [],
  "coreQuestions": [],
  "mustCover": [],
  "avoidDirections": [],
  "confirmedFacts": [],
  "uncertainItems": [],
  "usableSources": [
    {
      "sourceId": "S1",
      "title": "",
      "url": "",
      "reason": ""
    }
  ],
  "titleCandidates": [
    {
      "title": "",
      "reason": "",
      "risk": ""
    }
  ],
  "writerBrief": "",
  "writerContract": {
    "articleMission": "",
    "selectedTitle": "",
    "topicThesis": "",
    "targetReader": "",
    "readerPromise": "",
    "firstSectionFocus": "",
    "mustAnswer": [],
    "mustCover": [],
    "mustNotDo": [],
    "confirmedFacts": [],
    "uncertainItems": [],
    "sourceBoundaries": [],
    "recommendedStructure": [],
    "tone": ""
  },
  "notes": []
}
```

### 루프 검증 단계

1. 최초 실행에서는 검색 후보 없이 판단한다.
2. 검색이 필요하면 `searchNeed`를 `light`, `normal`, `strict` 중 하나로 반환하고 `REVISION`을 반환한다.
3. 앱은 이 결과를 보고 Search Collector를 실행한다.
4. 앱은 최초 Research 결과를 `research-title-initial-result.json`으로 보존한다.
5. 검색 후보와 sourceQuality를 포함해 Research/Title Agent를 재실행한다.
6. 재실행 후 `searchNeed`가 유효하지 않으면 `failurePhase: research`로 실패 처리한다.
7. `searchNeed`가 검색 필요인데 검색 후보가 없으면 `failurePhase: research`로 실패 처리한다.
8. `status`가 `BLOCK`이면 본문 작성으로 넘어가지 않는다.
9. `status`가 `REVISION`이면 본문 작성으로 넘어가지 않는다.
10. `status`가 `PASS`여도 `finalTitle`이 비어 있으면 실패 처리한다.
11. writerContract는 검색 과정 설명이 아니라 Writer가 그대로 따라야 할 작성 계약이어야 한다.

## 12. Search Collector

### 역할

Research/Title Agent가 검색을 요청할 때만 실행한다.

사용자가 입력한 카테고리, 키워드, 발행 목적, Research/Title Agent의 coreQuestions, mustCover, uncertainItems를 이용해 검색 후보를 수집한다.

### 검색 공급자

- 기본 공급자는 NAVER이다.
- 보조 공급자는 Google이다.
- 검색 URL 템플릿을 설정에서 바꿀 수 있어야 한다.

### 후보 처리

1. 검색 결과에서 제목, URL, 요약을 수집한다.
2. 네이버 블로그 URL은 모바일 본문 추출 URL로 변환할 수 있어야 한다.
3. 후보 본문을 가능한 범위에서 추출한다.
4. 공통 주제 토큰, 키워드 토큰, 제목/요약 관련도를 계산한다.
5. 저신뢰 도메인은 strictEvidence일 때 감점한다.
6. strictEvidence일 때 공식 도메인, 현재성 단서, 직접 주제 단서를 가점한다.
7. strictEvidence가 필요한데 강한 후보가 없으면 공식 도메인 중심 검색을 한 번 더 시도한다.

### strictEvidence 조건

strictEvidence는 전역 하드코딩이 아니라 필요한 카테고리와 키워드일 때만 작동해야 한다.

다음 조건 중 하나 이상이면 strictEvidence를 활성화한다.

- Research/Title Agent의 `searchNeed`가 `strict`이다.
- `searchNeed`가 `normal`이고 발행 목적 또는 리서치 지침에 모집, 신청, 접수, 공고, 지원, 정책, 자격, 기간, 현재 운영 같은 현재성 검증 단서가 있다.

### sourceQuality

검색 후 sourceQuality를 만든다.

```json
{
  "status": "strong | weak | insufficient | skipped | unknown",
  "reason": "",
  "candidateCount": 0,
  "strongEvidenceCandidates": 0
}
```

strictEvidence일 때는 공식 또는 신뢰 가능한 현재성 후보가 없으면 `insufficient`를 반환한다.

## 13. Writer Agent

### 역할

Research/Title Agent가 만든 `finalTitle`과 `writerContract`만을 기준으로 본문을 작성한다.

Writer Agent는 새 주제를 만들거나 제목을 바꾸면 안 된다.

Writer Agent는 검색 과정, 검증 과정, 출처 수집 과정을 독자에게 설명하면 안 된다.

### 출력 JSON

```json
{
  "status": "success | failed",
  "failureReason": "",
  "title": "",
  "article": "",
  "tags": [],
  "titleImagePath": "",
  "titleImagePrompt": "",
  "bodyImages": [
    {
      "sequence": 1,
      "path": "",
      "prompt": ""
    }
  ],
  "notes": []
}
```

### 본문 규칙

- 본문은 한국어로 작성한다.
- `[SECTION - 제목]` 마커를 사용해 섹션을 구분한다.
- 글의 첫 문단은 독자가 얻을 내용을 바로 설명해야 한다.
- 발행 목적을 그대로 복사하거나 “이 글은 일반 조언이 아니라” 같은 방어적 문장으로 시작하지 않는다.
- factBased 주제는 Research/Title Agent가 넘긴 confirmedFacts와 usableSources 범위 안에서만 쓴다.
- 불확실한 항목은 단정하지 않고 확인 필요 항목으로 쓴다.
- 날짜, 작성일, 오늘 날짜, 현재 기준일을 노출하지 않는다. 단, 출처가 확인한 공식 기준일 또는 공고 기간은 쓸 수 있다.
- 소스 문장을 그대로 복사하지 않는다.
- 이미지 생성은 직접 하지 않고 프롬프트만 작성한다.

### 이미지 프롬프트 규칙

- `titleImagePrompt`는 전체 글의 대표 썸네일 프롬프트이다.
- 본문 이미지 프롬프트는 해당 `[IMAGE INSERT - n]` 주변 섹션 내용을 요약해야 한다.
- 제목 이미지는 짧은 한글 텍스트를 포함할 수 있다.
- 본문 이미지는 텍스트를 과도하게 넣지 않는다.
- 계정별 `accountImageStylePrompt`가 있으면 사실성과 정책보다 낮은 우선순위로 적용한다.

### 루프 검증 단계

1. Writer Agent 실행 후 `status`가 `success`인지 확인한다.
2. `article`이 비어 있으면 실패 처리한다.
3. `tags`가 배열이 아니면 실패 처리한다.
4. 작성일, 오늘 날짜, 현재 날짜, 기준일 누출이 있으면 재시도 가능한 실패로 처리한다.
5. Research/Title Agent가 `REVISION` 상태였거나 소스 부족을 이유로 Writer가 실패하면 재시도 가능한 실패로 처리한다.
6. 자동 주제 모드에서는 Main Agent 검수와 묶어 최대 3회까지 재작성할 수 있다.
7. 수동 주제 모드에서는 기본 1회만 시도한다.
8. 재시도 시에는 Main Agent 또는 Writer 실패 이유를 `writerRevisionFeedback`으로 넘긴다.
9. 재시도 Writer는 기존 제목과 writerContract를 유지하고 본문 JSON만 수정한다.

## 14. Main Agent 최종 검수

### 역할

Writer Agent 결과가 실제 발행 가능한지 독립적으로 판단한다.

Main Agent는 Writer의 success를 그대로 믿으면 안 된다.

### 출력 JSON

```json
{
  "status": "PASS | REVISION | BLOCK",
  "failureReason": "",
  "titleReviewPass": true,
  "articleAnswersTitle": true,
  "topicPreserved": true,
  "factualityPass": true,
  "sourceUsePass": true,
  "bodyQualityPass": true,
  "riskExpressionPass": true,
  "writerContractPass": true,
  "readerFacingArticlePass": true,
  "noResearchProcessNarrationPass": true,
  "publishable": true,
  "issues": [],
  "revisionInstructions": [],
  "notes": []
}
```

### 검수 기준

- 제목이 Research/Title Agent의 finalTitle과 맞아야 한다.
- 본문이 제목에 답해야 한다.
- writerContract의 articleMission, topicThesis, readerPromise, firstSectionFocus, mustAnswer, mustCover, mustNotDo를 지켜야 한다.
- factBased 주제는 확인된 사실 범위 안에서만 작성되어야 한다.
- 공식 또는 신뢰 가능한 출처가 필요한 주제인데 근거가 없으면 PASS할 수 없다.
- 모집, 신청, 접수, 현재 운영, 정책 조건, 가격, 일정, 마감일은 현재성 검증이 있어야 한다.
- 글 첫머리가 검색 과정 설명처럼 보이면 REVISION이다.
- 발행 목적 문장을 그대로 복사하면 REVISION이다.
- `[SECTION - ...]` 마커는 앱 내부 섹션 마커로 허용한다.
- 정책, 지원, 모집, 교육 주제는 대상, 조건, 신청/확인 경로, 변동 가능 항목, 주의사항을 실용적으로 다뤄야 한다.

### 루프 검증 단계

1. Main Agent 실행 후 `status`를 확인한다.
2. `PASS`라면 모든 boolean 필드가 true인지 앱에서 다시 확인한다.
3. `PASS`인데 하나라도 false이거나 failureReason이 있으면 PASS 검증 실패로 처리한다.
4. `PASS`이고 모든 필드가 true이면 Writer/Main 루프를 종료한다.
5. `REVISION`이거나 PASS 검증 실패이면 자동 주제 모드에서 Writer Agent를 재실행한다.
6. 재실행 가능 횟수가 남아 있으면 `revisionInstructions`, `issues`, PASS 검증 실패 이유를 Writer에게 전달한다.
7. 재실행 가능 횟수가 없으면 `failurePhase: main_review`로 실패 처리한다.
8. `BLOCK`이면 즉시 실패 처리하고 다음 단계로 넘어가지 않는다.

## 15. Image Worker

### 역할

Writer Agent가 만든 `titleImagePrompt`와 `bodyImages[].prompt`를 이용해 Codex 이미지 생성을 실행한다.

Image Worker는 본문, 제목, 태그, 사실관계를 수정하지 않는다.

### 출력 JSON

```json
{
  "status": "success | failed",
  "failureReason": "",
  "titleImagePath": "",
  "bodyImages": [
    {
      "sequence": 1,
      "path": "",
      "prompt": ""
    }
  ],
  "notes": []
}
```

### 이미지 저장 규칙

- 이미지 도구가 실제 로컬 파일 경로를 반환하면 그 경로를 JSON에 적는다.
- 이미지 도구가 base64 또는 세션 데이터로 결과를 남기면 JSON에 base64를 붙여 넣지 않는다.
- 이 경우 path를 비워 두고, 앱이 Codex 세션 JSONL에서 이미지 데이터를 읽어 파일로 저장한다.
- 로그는 “경로 반환 실패”가 아니라 “이미지 생성 완료, 파일 저장 중입니다”로 표시한다.
- Image Worker는 앱 image 폴더로 직접 복사하지 않는다.

### 루프 검증 단계

1. Image Worker는 이미지 생성 범위를 확인한다.
2. 제목 이미지가 켜져 있으면 제목 이미지 1개를 생성한다.
3. 본문 이미지 수 제한을 지킨다.
4. 로컬 경로가 있으면 앱이 해당 파일을 `runtime/image/`로 복사한다.
5. 로컬 경로가 없고 이미지 요청이 있었으면 앱이 Codex 세션에서 `image_generation_end` 또는 `image_generation_call`의 base64 PNG/JPG/WEBP 결과를 찾아 `jobDir/recovered-images/`에 저장한다.
6. 복구된 이미지를 `runtime/image/<safeTopic>_img_title.png` 또는 `runtime/image/<safeTopic>_img_1.png`로 복사한다.
7. 복구 성공 시 “Codex 이미지 결과 n개를 파일로 저장했습니다”를 info 로그로 남긴다.
8. 이미지 생성 또는 저장이 실패해도 본문 작성 결과는 유지하고 이미지 없는 발행이 가능해야 한다.

## 16. 이미지 정규화와 미리보기

`imageAssets` 모듈을 구현한다.

기능은 다음과 같다.

- Writer/Image Worker가 반환한 이미지 경로를 절대 경로로 해석한다.
- 이미지 파일 확장자는 `.png`, `.jpg`, `.jpeg`, `.webp`만 허용한다.
- body image는 sequence 순서로 처리한다.
- 이미지 파일이 없으면 경고 노트를 남긴다.
- Codex 세션 base64 이미지 결과를 복구한다.
- 복구 이미지는 `jobDir/recovered-images/codex-image-n.<ext>`로 저장한다.
- 최종 미리보기에는 `file://` URL을 만들어 렌더러로 보낸다.
- 본문 이미지가 생기면 `[IMAGE INSERT - n]` 마커를 보장한다.

## 17. 제목 중복 검사

작업 이력은 `runtime/blog_history.jsonl`에 저장한다.

제목 중복 검사를 위해 로컬 해시 임베딩을 구현한다.

규칙은 다음과 같다.

- 계정의 blogId별 과거 제목만 비교한다.
- 새 제목과 기존 제목의 cosine similarity를 계산한다.
- 유사도 0.75 이상이면 `duplicate_retry` 상태로 중단한다.
- 이 경우 발행하지 않고 이력에 기록한다.

## 18. 자동 발행 루프

자동 발행은 렌더러에서 제어한다.

### 대상 순회

1. 체크된 계정만 사용한다.
2. 체크된 카테고리만 사용한다.
3. 카테고리명과 키워드가 모두 있는 대상만 자동 발행 대상으로 삼는다.
4. 현재 계정의 다음 카테고리로 이동하되, 세션 만료 시 가능하면 다른 계정으로 넘어간다.

### 자동 Cycle 재시도 규칙

1. 일반 실패는 최대 3회까지 같은 대상으로 재시도한다.
2. Research 단계 실패는 최대 2회까지만 같은 대상으로 재시도한다.
3. 실패 결과의 `failurePhase`가 `research`이면 Research 실패로 본다.
4. `duplicate_retry`는 재시도 대상이 아니다.
5. `session_expired`는 세션 확인 또는 반복 주기까지 대기한다.
6. `codex_usage_limit`은 자동 발행 전체를 중단한다.
7. 재시도 한도에 도달하면 다음 대상으로 넘어간다.

## 19. 네이버 발행 자동화

`naverPublisher` 모듈을 구현한다.

### Chrome과 세션

- Playwright Core로 사용자의 Chrome을 persistent context로 실행한다.
- 계정별 브라우저 프로필 폴더를 분리한다.
- Chrome profile의 비정상 종료 표시를 정리하는 helper를 둔다.
- 로그인 화면이면 네이버 ID와 비밀번호를 입력한다.
- 자동 발행에서 로그인 필요 상태가 발생하면 `SESSION_EXPIRED`로 처리할 수 있어야 한다.
- 세션 확인 모드에서는 글쓰기 편집기 접근 가능 여부를 확인하고 계정 상태를 `valid` 또는 `expired`로 갱신한다.

### 글쓰기 입력

1. `https://blog.naver.com/<blogId>/postwrite`로 이동한다.
2. 기존 작성 글 팝업이 있으면 닫거나 대기한다.
3. 제목 입력 영역이 보일 때까지 기다린다.
4. 제목을 입력한다.
5. 제목을 최상단 인용구 블록으로 한 번 더 삽입한다.
6. 제목 이미지가 있으면 삽입하고 AI 활용 설정을 켠다.
7. 본문을 `[SECTION - ...]` 마커 기준으로 섹션 제목과 문단으로 나누어 입력한다.
8. `[IMAGE INSERT - n]` 위치에 본문 이미지를 삽입한다.
9. 각 이미지 삽입 후 AI 활용 설정 토글을 확인하고 켠다.
10. 태그를 입력한다.

### AI 활용 설정 토글

AI 활용 설정은 다음 DOM을 기준으로 찾는다.

```html
<button type="button" class="se-set-ai-mark-button-toggle"></button>
<button type="button" class="se-set-ai-mark-button-toggle se-is-selected"></button>
```

검증 규칙은 다음과 같다.

- `button.se-set-ai-mark-button-toggle.se-is-selected`이면 이미 켜진 상태이다.
- 부모 `.se-set-ai-mark-button.se-is-selected`도 켜진 상태로 인정한다.
- `aria-pressed="true"` 또는 `aria-checked="true"`도 켜진 상태로 인정한다.
- 꺼진 토글을 찾으면 클릭해 켠다.
- 클릭 후 `se-is-selected`가 붙는지 확인한다.
- 실패하면 한 번 더 클릭한다.
- 그래도 확인되지 않으면 경고 로그를 남기고 발행 흐름은 계속한다.

### 발행 버튼 선택

편집기 상단 발행 설정 버튼은 class에 의존하지 않는다.

우선순위는 다음과 같다.

```text
button[data-click-area='tpb.publish']
[data-click-area='tpb.publish']
button[data-click-area*='publish']:text-is('발행')
[data-click-area*='publish']:text-is('발행')
button:has(span:text-is('발행'))
button:text-is('발행')
[role='button']:text-is('발행')
[role='button']:has-text('발행')
```

`예약 발행 1건`처럼 `발행`을 포함하는 다른 버튼을 먼저 누르지 않도록 selector 배열을 순서대로 시도한다.

### 발행 설정

- 공개 범위는 `private` 또는 `public`을 지원한다.
- 즉시 발행과 예약 발행을 지원한다.
- 예약 발행은 현재 시각에서 지정 시간 뒤를 계산하고 10분 단위로 반올림한다.
- 카테고리 선택이 있으면 선택한다.
- 최종 발행 버튼은 팝업/다이얼로그 내부 버튼을 우선 찾는다.
- 발행 완료 문구나 postwrite를 벗어난 blog URL로 완료를 판단한다.

7. Research/Title Agent가 검색 필요성을 판단하고 검색 후보 수집 후 재분석한다.
8. Writer Agent와 Main Agent가 자동 주제 모드에서 최대 3회 루프를 돈다.
9. Research 실패는 자동 발행에서 최대 2회 후 다음 대상으로 넘어간다.
10. 일반 실패는 자동 발행에서 최대 3회 후 다음 대상으로 넘어간다.
11. 이미지가 base64 세션 결과로만 생성되어도 앱이 파일로 저장한다.
12. 발행 버튼 클릭 시 `예약 발행 1건`이 아니라 실제 `발행` 버튼을 클릭한다.
13. 이미지 삽입 후 AI 활용 설정 토글을 켠다.
14. 제목 중복 유사도 0.75 이상이면 발행하지 않는다.
15. `npm run check`가 통과한다.
16. `runtime/`, `dist/`, 계정 JSON, env, exe가 Git에 올라가지 않는다.

## 26. 구현 시 주의사항

사용자 입력과 계정 데이터는 로컬 runtime에만 둔다.

코딩 에이전트는 하드코딩된 카테고리별 사이트 목록을 만들지 말고, Research/Title Agent의 `searchNeed`와 발행 목적, 키워드, 리서치 지침을 보고 필요한 경우에만 강한 출처 검증을 켠다.

공식성, 주제 직접성, 현재 모집/신청 단서는 모든 카테고리에 전역 적용하지 않는다.

이미지 생성은 OpenAI 이미지 API를 앱에서 직접 호출하지 않고 Codex Image Worker를 통해 수행한다.

Codex 이미지 결과가 파일 경로가 아니라 세션 base64로 남는 것은 정상 흐름으로 취급한다.

토큰 사용량은 단순 누적 gross만 보여주지 말고 캐시 입력을 제외한 실효 토큰, gross 토큰, agent별 합계를 함께 계산한다.

발행 자동화는 실패해도 브라우저 세션과 계정 상태를 보존해야 하며, 세션 만료는 사용자가 다시 로그인할 수 있게 대기 흐름으로 처리한다.
